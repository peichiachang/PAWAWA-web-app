# 食物紀錄計算問題評估：250ml 碗、顯示 80g、邊界區過大

## 為何會出現「約 90g」？

- 介面顯示的 **250** 是 **容量（volumeMl = 250 ml）**，不是「滿碗幾克」。
- 程式用公式把容量換算成參考克數：**volumeMl × 0.8 × 0.45 = 250 × 0.8 × 0.45 = 90g**  
  （0.8 = 假設約八分滿，0.45 = 乾飼料密度 g/ml）
- 若您的碗**實際裝滿約 250g**，應在食碗校準裡填寫 **「滿碗克數」= 250**；有填時會用 250g 當放飯參考與 AI 錨定，不再用 90g。

---

## 一、現象整理

- **介面**：選擇「白色貓碗 (250ml)」、乾飼料；AI 分析結果顯示 **Household: 80g**、**AI 估算攝取: 80g**；**邊界區需人工確認 (ratio=0.650)**。
- **使用者預期**：該碗設定為 250（單位為 250ml 或 250g 需釐清），結果出現 80g，與預期落差大；邊界區需人工確認的範圍極大。

---

## 二、問題根因分析

### 1. 單位與換算：250ml ≠ 250g

- 介面顯示為 **「250ml」**（容量），不是「250g」。
- 程式內「放飯量（參考）」的計算為：
  - `t0RefGramsForBoundary = volumeMl * 0.8 * 0.45`
  - 250ml → **250 × 0.8 × 0.45 = 90g**（約滿碗乾飼料克數）。
- 因此：
  - 若使用者把「250ml」理解成「250g」，會覺得 80g 與 250 差很多。
  - 若碗的校準真的是「滿碗約 250g」，但目前存的是 **volumeMl = 250**，則系統會用 90g 當參考，與「250g 滿碗」的預期不符。

**結論**：需要釐清「該碗設定是 250g」是指容量 250ml，還是期望滿碗 250g；並在 UI 或校準流程區分 ml（容量）與 g（克數）。

---

### 2. 後端 API 未收到碗容量／參考克數（最主要技術原因）

- **Production（Vercel）** 使用 **HTTP 模式**，呼叫 `/api/ai/feeding`。
- 前端送出的 payload **只有**：
  - `t0ImageRef`, `t0ImageBase64`, `t0MimeType`
  - `t1ImageRef`, `t1ImageBase64`, `t1MimeType`
- **沒有傳**：
  - `vessel.volumeMl`
  - `t0.manualWeight`
  - 任何「滿碗約幾克」的參考（例如 t0RefGrams 或 defaultPortionGrams）

因此後端 prompt 只給模型兩張圖，**沒有告訴模型「這個碗滿載大約多少克」**。模型只能從影像猜「吃了多少克」，容易與真實碗容量脫節：

- 可能猜出 80g（你看到的值），而與前端用 250ml 換算的 90g 不一致。
- **consumedRatio**（例如 0.65）與 **consumedGram**（80）可能也不一致：若滿碗為 90g，吃掉 80g 應約 0.89，不是 0.65；若模型心裡的「滿碗」是別的数字，就會出現這種落差。
- 邊界區是依 **consumedRatio** 觸發（見下），ratio 不準會直接導致「邊界區很大」的感受。

**結論**：**80g 完全來自模型只看圖的猜測，沒有錨定到「此碗 250ml ≈ 約 90g 滿碗」的資訊。** 要讓結果合理，必須把「碗容量或參考克數」傳給後端並納入推論。

---

### 3. 邊界區觸發條件與 ratio=0.650

- 邊界區條件（FeedingModal）：
  - `isBoundaryRatio = |consumedRatio - 0.375| <= 0.07 || |consumedRatio - 0.625| <= 0.07`
- **ratio=0.650** 落在 **0.625 ± 0.07**（約 0.555～0.695）區間，因此會進入「邊界區需人工確認」。
- 設計原意是：在「約 1/3 或約 2/3 吃掉」的不確定帶，要求使用者手選攝取程度。
- 若模型經常回傳 0.6～0.7 的 ratio（例如對淺碗、特殊形狀碗不確定），就會**經常**觸發邊界區，造成「邊界區範圍極大」的體感。

**結論**：邊界區邏輯本身符合設計，但因為 **consumedRatio 未與碗的參考克數對齊**，容易出現「顯示 80g 卻 ratio 0.65」這種不一致，並讓邊界區感覺過大。

---

### 4. 特殊碗型（防漏／黃色凸起）對視覺的影響

- T0 照片為**貓耳白碗、中央黃色防漏凸起**，飼料堆在周圍與上方。
- 若校準時用的是「空碗幾何容量 250ml」，**實際可放飼料的體積**會小於 250ml（被黃色凸起佔掉一部分）。
- 模型可能：
  - 把黃色區域當成「非飼料」而只估「可見飼料體積」→ 克數偏低（例如 80g）；
  - 或對「滿碗」的定義與我們 90g 不一致，導致 ratio 與克數都不準。

**結論**：特殊碗型會放大「只看圖、沒有碗容量／參考克數」時的誤差；仍應以「傳入參考克數／容量」為主要解法，必要時再在 prompt 註明「碗內可能有障礙物，請只估飼料體積」。

---

## 三、與預期算式的落差

- 前端「放飯量（參考）」：**250 × 0.8 × 0.45 = 90g**（滿碗約 90g）。
- 攝取程度選項：「吃完了」約 90g、「吃了一半」約 45g 等，都是以 90g 為基準。
- 但 **AI 回傳 80g、ratio 0.65**：
  - 若以 90g 為滿碗，80g 應接近「幾乎吃完」，ratio 應約 0.89，不是 0.65。
  - 顯示 80g 卻又要求「邊界區人工選 23g / 45g / 68g」，會讓使用者覺得與「約 90g 吃完了」的預期差很多。

落差來自：**後端沒有 90g（或 250ml）這個錨點，模型輸出的克數與比例未與前端參考一致。**

---

## 四、建議修正方向（不改變既有算式與辨識流程的前提下）

### 1. 後端：傳入「參考滿碗克數」並納入 prompt（優先）

- 前端在 HTTP 模式送 `/ai/feeding` 時，一併傳送：
  - `volumeMl`（可選）  
  - 或 **`t0RefGrams`**（建議）：即目前的 `t0RefGramsForBoundary`（manualWeight 或 volumeMl * 0.8 * 0.45）。
- 後端在 prompt 中明確寫入：
  - 「此碗滿載約 **X** 克乾飼料，請估計本次吃掉的克數（0～X）及吃掉的比例（0～1）。」
- 如此模型回傳的 **consumedGram** 與 **consumedRatio** 會與「約 90g 滿碗」對齊，減少 80g vs ratio 0.65 的不一致，邊界區觸發也會更合理。

### 2. 前端：HTTP 請求帶上 vessel／t0RefGrams

- 在 `app/src/services/ai/index.ts` 的 `analyzeFeedingImages` 中，對 `/ai/feeding` 的 body 增加：
  - `volumeMl: input.vessel?.volumeMl ?? null`
  - `t0RefGrams: input.t0.manualWeight ?? (input.vessel?.volumeMl ? input.vessel.volumeMl * 0.8 * 0.45 : null)`  
  或由呼叫端算好再傳 `t0RefGrams`。
- 後端 `api/ai/feeding.js` 或 `api/_lib/ai.js` 讀取 `body.t0RefGrams` 或 `body.volumeMl`，並寫入 prompt（如上）。

### 3. UI：釐清 250ml 與「約 90g」的關係

- 在食碗選項或說明中，必要時顯示：「250ml（滿碗約 90g 乾飼料）」或類似文案，避免「250」被解讀成 250g。
- 若校準流程允許使用者輸入「滿碗克數」，應區分「容量(ml)」與「預設滿碗克數(g)」，並在計算與顯示上一致使用。

### 4. 邊界區（可選優化）

- 若已傳入 `t0RefGrams` 且後端回傳的 **consumedRatio** 與 **consumedGram** 一致（例如 consumedGram ≈ consumedRatio * t0RefGrams），可考慮在邊界區內用 **ratio × t0RefGrams** 當顯示克數，減少「80g 卻要選 23/45/68g」的違和感。
- 或維持現有邊界區邏輯，待參考克數傳遞上線後，再依實際 ratio 分布考慮是否縮小 0.375／0.625 的區間。

---

## 五、小結

| 項目 | 說明 |
|------|------|
| **250 與 80 的落差** | 介面為 250**ml**，系統換算滿碗約 **90g**；80g 來自模型只看圖、**未收到碗容量／參考克數** 的估計。若使用者以為是 250g，落差會更大。 |
| **Household / AI 估算皆 80g** | 後端只回傳一組 consumedGram，前端用於「Household」與「AI 估算攝取」，故兩者相同；問題在於該值未錨定到 90g。 |
| **邊界區極大** | ratio=0.650 落在 0.625 邊界帶；因 ratio 與克數未與 90g 對齊，常觸發邊界區且與「約 90g 吃完了」的預期不符。 |
| **建議優先** | **前端傳 volumeMl 或 t0RefGrams，後端在 prompt 中寫入「滿碗約 Xg」，讓模型輸出與 90g 對齊。** 再輔以 UI 釐清 250ml vs 約 90g。 |

以上評估可作為實作「傳參考克數＋prompt 錨定」與 UI 說明的依據；未改動既有體積／密度算式與辨識流程，僅補上「碗容量／參考克數」的傳遞與使用。
